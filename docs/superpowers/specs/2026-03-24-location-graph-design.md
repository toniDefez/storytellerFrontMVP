# Location Graph — Design Spec

**Date:** 2026-03-24
**Status:** Approved (brainstorming session with Narratologist, Game Designer, UX Architect, Backend Architect, Anthropologist)

---

## Vision

The location graph is the bridge between the abstract world (causal graph) and the concrete story (characters, scenes). It answers the question every storyteller eventually asks: *where does this happen?*

The model is Dwarf Fortress: depth lives in the generation, not in the UI. The AI derives locations from the causal graph's properties — if the world has scarce water, settlements cluster around water nodes. The user sees a clean map. The richness is implicit.

Core principle: **given the right tools, anyone can create something unique.**

---

## What It Is

A second interactive graph on the World Detail page. Nodes are places (settlements, wildernesses, ruins, landmarks). Edges are travel connections (roads, rivers, dangerous passes). The graph is generated by AI using the causal graph as premise, then the user can add, edit, and connect nodes manually.

**It is not a map.** It is a spatial argument: each location exerts pressure on the characters and scenes that happen there. The AI derives that pressure from the world's causal properties.

---

## Scope

### In scope (v1)
- AI generation of the full location graph from causal graph context
- Manual node creation, editing, deletion
- Manual edge creation (drag between nodes), editing, deletion
- Tab switcher in WorldDetailPage (Causal ↔ Localizaciones)
- Node dragging with backend persistence of canvas positions
- Node detail panel with optional "story layer"
- Regenerate action with confirmation dialog

### Out of scope (future)
- Linking scenes and characters to location nodes (FK on scene/character entities)
- Location-aware scene generation
- Sub-graphs within a location (street-level, room-level)
- Location state changes driven by story events

## Visual Design

**Canvas background:** `hsl(180 8% 97%)` — subtly cooler than the causal graph's warm parchment, just enough to register the shift.

**Node accent bar:** teal-500 constant (no domain colors — location nodes don't have causal domains).

**Edges:** `bezier` type, no arrowhead (bidirectional by default), dashed stroke at 60% teal-500 opacity. Directional edges (one-way paths) use a single arrowhead.

**Color token:** `#14b8a6` (teal-500). Does not conflict with purple (worlds), orange (characters), or cyan (scenes).

## Node Schema

### Visible by default
| Field | Type | Notes |
|-------|------|-------|
| `name` | string | The place's name |
| `node_type` | enum | settlement / wilderness / ruin / landmark / threshold |
| `description` | text | 2–3 sentences. AI-generated. Sensory texture + who's here. |
| connections | derived | Edge list shown as pills |

**`threshold`** is a distinct type (bridges, gates, border towns, mountain passes). These locations exist to be crossed, not inhabited — they generate dramatic tension through passage.

### Optional "Story Layer" (collapsed by default)
| Field | Type | Notes |
|-------|------|-------|
| `atmosphere` | text | Sensory anchor: what you smell, hear, or see on arrival |
| `social_filter` | text | Who belongs here and who doesn't |
| `behavioral_rule` | text | What you can / cannot do here |
| `control` | text | Who controls this place and how you can tell |
| `duration` | enum | Permanent / Temporary / Declining / Emerging |

These fields are AI-populated during generation and user-editable. They are never required upfront. Power users find them; new users ignore them.

The story layer is the minimum to anchor a scene (per anthropological analysis): one sensory anchor + one behavioral constraint makes a generic setting specific.

---

## Edge Schema

Two orthogonal dimensions, 9 configurations total:

**Effort** (how hard to traverse): `easy` / `moderate` / `difficult`

**Type** (what kind of movement): `road` / `wilderness` / `waterway`

Plus:
- `bidirectional: boolean` (default true)
- `note: string` — free text for specifics ("closes in winter", "guarded by the Crown")

This avoids taxonomy overhead while covering the meaningful range. A difficult waterway is rapids in a storm. An easy road is a royal highway.

---

## Graph Structure

Generated graphs follow a **clusters-with-bottlenecks** topology:

- **Tier 1 — Poles (3–5):** Ideological anchors. Each embodies a different answer to the world's central tension. Characters must choose between them.
- **Tier 2 — Active nodes (6–12):** Where story happens. Factions visible, conflicts active. AI generates these in the first pass.
- **Tier 3 — Waypoints (5–15):** Transit spaces. Structurally necessary for pacing, narratively thin.

**Total: 12–30 nodes** for a complete world. Below 12 feels like a board game. Above 30 creates open-world sprawl.

Cross-cluster connections (the bottlenecks) are where interesting things happen — border towns, disputed passes, the only port. These are deliberately sparse and structurally significant.

## Generation Flow

### Trigger
- **First visit (no nodes):** Empty state with a single primary button: "Generar desde el grafo causal". Disabled if causal graph has 0 nodes, with tooltip explaining the dependency.
- **Regeneration:** "Regenerar" button in graph toolbar → confirmation dialog (replaces existing graph) → generates.

### Generation pipeline
Same RabbitMQ `PublishAndWait` pattern as all other generation endpoints.

**Important:** The `worlds` table does NOT have flat columns for environment/subsistence/organization/tensions/tone. The rich world context lives in `world_nodes` (the causal graph tree). The Go handler fetches all world_nodes for the world, organizes them by domain, and builds the payload.

**New payload type (`LocationGeneratePayload`):**
```go
type LocationNodesByDomain struct {
    Label       string `json:"label"`
    Description string `json:"description"`
    Role        string `json:"role"` // state / event / rupture
}

type LocationGeneratePayload struct {
    WorldID       int                                   `json:"worldId"`
    Premise       string                                `json:"premise"`      // worlds.premise
    NodesByDomain map[string][]LocationNodesByDomain   `json:"nodesByDomain"` // keyed by domain
    NodeCountHint int                                   `json:"nodeCountHint"`
}
```

`NodesByDomain` keys: `core`, `physical`, `biological`, `social`, `symbolic`, `technic`. The Go handler queries `world_nodes WHERE world_id = $1` and groups by `domain`. This is the Dwarf Fortress model: generate locations from the causal properties, not from abstract labels.

**Generator instructions:**
- Derive settlement positions from `physical` and `biological` nodes (where can people live given geography + ecology)
- Derive edge types from `physical` nodes (rivers → waterway edges, mountains → difficult wilderness edges)
- Derive dangerous zones from `social` + `symbolic` nodes (contested factions, taboo territories)
- Derive cluster structure from power dynamics in `social` nodes
- Populate `atmosphere` with sensory detail derived from `physical` + `biological` nodes
- Populate `social_filter` and `behavioral_rule` from `social` + `symbolic` nodes
- Return canvas coordinates as rough geographic layout (x: 0–800, y: 0–600 normalized)
- Return strict JSON, no prose

**Generator response:**
```json
{
  "nodes": [
    {
      "name": "Puerto de Arenmar",
      "node_type": "settlement",
      "description": "Ciudad costera que huele a sal y aceite de ballena quemado.",
      "atmosphere": "Olor a fruta podrida y brasas en el mercado del puerto",
      "social_filter": "Los marineros duermen aquí. Los mercaderes pasan pero no se quedan.",
      "behavioral_rule": "No se habla de los naufragios del año pasado.",
      "control": "El Gremio de Marineros controla qué barcos atracan.",
      "duration": "permanent",
      "canvas_x": 620,
      "canvas_y": 180
    }
  ],
  "edges": [
    {
      "source_name": "Puerto de Arenmar",
      "target_name": "El Paso Salado",
      "edge_type": "road",
      "effort": "moderate",
      "bidirectional": true,
      "note": ""
    }
  ]
}
```

Edges reference nodes by name. Go handler inserts nodes first, builds name→id map, then inserts edges. Same pattern as causal graph.

## ReactFlow Implementation (@xyflow/react v12)

### Key architectural difference from the causal graph

| | Causal graph | Location graph |
|---|---|---|
| `nodesDraggable` | `false` | `true` |
| Layout | `computeTreeLayout()` (fixed tree) | AI-provided `canvas_x/y`, free positioning |
| Edges | Fixed handles (top/bottom) | **Floating edges** |
| `connectionMode` | default (Strict) | `ConnectionMode.Loose` |
| `onConnect` | — | opens edge-type dialog |
| `onEdgeClick` | — | selects edge → side panel |
| `onNodeDragStop` | — | debounced PATCH position |

### Floating edges

Location edges use the **floating edge** pattern (`useInternalNode()` + geometric intersection). Unlike the causal graph which has fixed top/bottom handles, floating edges connect to the nearest point on the node's border based on the relative position of the two nodes. A node to the right connects from the right side. A node above connects from the top. This gives the map its natural geographic feel without requiring explicit 4-sided handles.

### Custom edge types

Three custom edge components, all based on `BaseEdge` + `getBezierPath`:

```
WaterwayEdge:  stroke="#14b8a6"  strokeDasharray="8,5"  no arrowhead
WildernessEdge: stroke="#92400e"  strokeDasharray="5,7"  no arrowhead
RoadEdge:      stroke="#14b8a6"  solid                   arrowhead if bidirectional=false
```

All edges:
- Use `EdgeLabelRenderer` to render an HTML badge showing `effort` level (easy/moderate/difficult) at the edge midpoint
- Increase `strokeWidth` and add glow when `selected=true`
- Add `MarkerType.ArrowClosed` as `markerEnd` when `data.bidirectional === false`

### onConnect validation

```ts
const onConnect = useCallback((connection: Connection) => {
  if (connection.source === connection.target) return        // no self-loops
  if (edgeAlreadyExists(connection.source, connection.target)) return  // no duplicates
  openEdgeFormDialog(connection)  // user picks edge_type + effort
}, [edges])
```

## Interaction Model

### Nodes
- **Draggable** — position persisted to backend on drag end (debounced 500ms PATCH)
- No explicit handles needed — floating edges handle connection geometry
- Right-click context menu: Edit / Add connection / Delete

### Edges
- User-drawn by dragging from any point on a node border to another node (`ConnectionMode.Loose`)
- Independently selectable via `onEdgeClick` — opens `LocationEdgeDetailPanel` in side panel
- Deletable via `Delete` key or side panel action
- No arrowhead by default (bidirectional); `MarkerType.ArrowClosed` when `bidirectional: false`

### Side panel
Dedicated `LocationSidePanel` (does not share with causal graph):
- When node selected: `LocationNodeDetailPanel` — name, type, description, connections list, story layer (collapsible), Edit / Delete actions
- When edge selected: `LocationEdgeDetailPanel` — effort, type, bidirectional toggle, note, Delete action
- When nothing selected: empty state with "Añadir localización" button

## Backend

### New tables

```sql
CREATE TABLE location_nodes (
  id          SERIAL PRIMARY KEY,
  world_id    INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  node_type   TEXT NOT NULL CHECK (node_type IN ('settlement','wilderness','ruin','landmark','threshold')),
  description TEXT,
  properties  JSONB NOT NULL DEFAULT '{}',
  canvas_x    REAL NOT NULL DEFAULT 0,
  canvas_y    REAL NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT now(),
  CONSTRAINT unique_location_name_per_world UNIQUE (world_id, name)
);

CREATE TABLE location_edges (
  id             SERIAL PRIMARY KEY,
  world_id       INTEGER NOT NULL REFERENCES worlds(id) ON DELETE CASCADE,
  source_node_id INTEGER NOT NULL REFERENCES location_nodes(id) ON DELETE CASCADE,
  target_node_id INTEGER NOT NULL REFERENCES location_nodes(id) ON DELETE CASCADE,
  edge_type      TEXT NOT NULL CHECK (edge_type IN ('road','wilderness','waterway')),
  effort         TEXT NOT NULL DEFAULT 'moderate' CHECK (effort IN ('easy','moderate','difficult')),
  bidirectional  BOOLEAN NOT NULL DEFAULT true,
  note           TEXT,
  created_at     TIMESTAMPTZ DEFAULT now(),
  CONSTRAINT no_self_loop CHECK (source_node_id <> target_node_id),
  -- For bidirectional edges: service layer normalizes so that source_node_id < target_node_id before insert.
  -- This prevents (A→B) and (B→A) coexisting for the same physical connection.
  CONSTRAINT unique_directed_edge UNIQUE (source_node_id, target_node_id)
);
```

The story layer fields (`atmosphere`, `social_filter`, `behavioral_rule`, `control`, `duration`) live in the `properties JSONB` column.

### Indexes
```sql
CREATE INDEX idx_location_nodes_world_id ON location_nodes(world_id);
CREATE INDEX idx_location_edges_world_id ON location_edges(world_id);
CREATE INDEX idx_location_edges_source   ON location_edges(source_node_id);
CREATE INDEX idx_location_edges_target   ON location_edges(target_node_id);
```

### API endpoints

All routes are `HandleSecure`. Every handler calls `httputils.GetUserID(r)` and verifies world ownership before returning or mutating data — same pattern as all existing handlers.

```
GET    /location/graph          ?world_id=   → { nodes: [], edges: [] }
POST   /location/nodes                       → LocationNode
PUT    /location/nodes/update   ?id=         → LocationNode
DELETE /location/nodes/delete   ?id=         → 204
PATCH  /location/nodes/position ?id=         → 204  body: { canvas_x, canvas_y }
POST   /location/edges                       → LocationEdge
PUT    /location/edges/update   ?id=         → LocationEdge
DELETE /location/edges/delete   ?id=         → 204
POST   /location/generate                    → { nodes: [], edges: [] }
```

`GET /location/graph` returns both nodes and edges in one round trip — critical for page load performance.

`PATCH /location/nodes/position` is a lightweight endpoint for drag persistence. Called on drag end, debounced 500ms on frontend.

### Generation endpoint logic
1. Receive `{ worldId, nodeCountHint }`
2. Fetch `worlds.premise` + all `world_nodes WHERE world_id = $1` from DB
3. Group world_nodes by domain → build `LocationGeneratePayload`
4. Publish via `PublishAndWait`
5. On response: begin transaction → insert nodes → build name→id map → resolve edge source/target names to IDs → insert edges (skip edges with unresolved names, log warning) → commit
6. Return full graph `{ nodes, edges }`

**Edge name resolution:** If the AI returns an edge whose `source_name` or `target_name` doesn't match any generated node, skip that edge and log a warning. Do not fail the transaction — a partial graph is better than no graph for a local 8B model that may occasionally hallucinate node names.

## Key Design Decisions

| Decision | Reason |
|---|---|
| Simple node (name + type + description) as default view | UX: nobody fills in fields they don't understand. Floor for 90% of use cases. |
| Story layer collapsed by default | Progressive disclosure. Power users find it; new users aren't intimidated. |
| 2-dimension edge typing (effort × type) | Covers meaningful range without taxonomy overhead. |
| Depth in generation, not in UI | Dwarf Fortress model: AI derives controlling_idea, social dynamics, etc. from causal graph. User sees clean map. |
| Nodes draggable + position persisted | Location graph is spatial. Users need to arrange their map. |
| Both canvases mounted simultaneously | Preserves ReactFlow viewport state when switching tabs. |
| scene.location stays as string for now | Connecting scenes/characters to location nodes is future work. Don't block v1 on it. |
| Clusters with bottlenecks topology | Most narratively generative structure: creates regions + meaningful cross-region travel. |
